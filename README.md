# UGC Polls System

Система пользовательских опросов (UGC) с высокой нагрузкой.  
Реализовано на **Django 5.2 + DRF**, упаковано в **Docker**, управление зависимостями – **Poetry**.

## 📋 Функциональные возможности

- Регистрация и аутентификация пользователей (Token Auth)
- Создание опросов (только автором) и их редактирование/удаление
- Прохождение опросов: последовательные вопросы, варианты ответов + возможность собственного текста
- Упорядочивание вопросов и вариантов ответов (поле `weight`)
- Автоматическая документация API (Swagger, ReDoc)
- Админ-панель с nested-редактированием опросов, вопросов, вариантов
- Агрегированная статистика: количество ответов, популярные ответы, среднее время прохождения
- Оптимизация под нагрузку: отдельные таблицы статистики, индексы, кеширование Redis
- Скрипт генерации тестовых данных (миллионы опросов с вопросами и ответами)
- Полное покрытие тестами (модели, сериализаторы, вьюхи, сервисы, сигналы)

## 🚀 Быстрый старт (локальная разработка)

### Требования

- Docker / Docker Compose
- Git

### Запуск

1. Клонировать репозиторий:

   ```bash
   git clone git@github.com:AlexeyKondrukevich/ugc-polls.git
   ```
   
   ```bash
   cd ugc-polls
   ```
2. Создать файл .env (скопировать из примера, отредактировать при необходимости (секретные ключи, пароли БД)):

	```bash
	cp .env.example .env
	```
	
3. Запустить контейнеры (локальный режим, runserver):

	```bash
	bash docker_compose.sh --build
	```
	или вручную:
	
	```bash
	docker-compose up --build
	```
4. Создать суперпользователя для админки:

	```bash
	docker-compose exec web python manage.py createsuperuser
	```
5. Открыть в браузере:

	- [API](http://localhost:8000/api/)
   - [Swagger](http://localhost:8000/api/swagger/)
   - [ReDoc](http://localhost:8000/api/redoc/)
   - [Админка](http://localhost:8000/admin/)

## 🌍 Развёртывание на сервере
1. Для продакшн-окружения используйте отдельный compose-файл и скрипт запуска.
2. Настройте Nginx для HTTPS и статики.

```bash
bash docker_compose_prod.sh --build
```

Отличия от локального режима:

- Вместо runserver используется Gunicorn (WSGI-сервер).
- Добавлен Nginx для раздачи статики и проксирования.
- Для асинхронной обработки (опционально) можно заменить Gunicorn на Uvicorn (см. комментарии ниже).


## 📈 Комментарии по оптимизации под нагрузку (ASGI и другое)

### ASGI вместо WSGI

- Заменить Gunicorn на Uvicorn или Daphne (изменить файл docker-compose.prod.yml).
- Переписать часто вызываемые эндпоинты (например, next-question) на async def с использованием async-методов ORM (например, Poll.objects.aget(...)).
- Для синхронного кода (например, сериализаторы) использовать sync\_to\_async.

### Кеширование

- Кешировать ответы next-question (с учётом токена пользователя) в Redis на 1-5 минут.
- Использовать cache_page с vary\_on\_headers('Authorization') (уже частично реализовано для списка опросов).
- Применить кеширование для статистики опроса.

### База данных

- Партиционирование таблицы UserAnswer по answered_at (например, по месяцам) для миллионов записей.
- Для статистики применять агрегатные таблицы (уже реализовано).

### Фоновые задачи

- Вынести обновление статистики из сигналов в Celery.
Это снизит нагрузку на основной цикл запроса и позволит обрабатывать большие объёмы ответов без блокировок.

### Мониторинг

- В продакшене использовать Sentry для отслеживания ошибок и Prometheus + Grafana для метрик.

### Тесты под нагрузкой

- Оптимизировать узкие места по результатам профилирования (например, с помощью django-silk)
- Использовать locust или wrk для симуляции тысяч одновременных пользователей.
